## Background: What Rummikub Difficulty Actually Means

In Rummikub, the player has a rack of tiles. The board contains existing sets (runs and groups). The player's task is to rearrange board tiles and place rack tiles such that all sets remain valid and all rack tiles are used.

What makes a Rummikub puzzle **genuinely hard** for a human:

1. **No tile has an obvious home.** The player cannot look at a rack tile and immediately see where it goes. Every placement requires displacing something else.
2. **Moves cascade.** Placing tile A requires moving tile B out of its set, which requires inserting tile C somewhere else, which requires splitting set D.
3. **The solution is non-obvious and unique.** There is exactly one arrangement that works, and finding it requires exploring many dead ends.
4. **Visual ambiguity.** Multiple board configurations look equally plausible. The human must mentally simulate several branches before finding the valid one.

What a computer solver cares about is entirely different: ILP column count, search tree size, constraint density. These do not map to human cognitive difficulty.

## Generation History

### v1: "Sacrifice" Approach (pre-phase4)

**Mechanism:**
1. Build a random valid board of N sets.
2. Pick K sets and remove them entirely from the board ("sacrifice").
3. Sample M tiles from the removed sets as the rack.
4. Solve to verify the rack can be placed.
5. Gate: reject if any rack tile can trivially extend any board set (`_any_trivial_extension`).

**Why it was better (in principle):**  
The sacrificed sets no longer exist. The rack tiles have no obvious home — their original context is gone. The player must find *new* arrangements. The `_any_trivial_extension` gate ensured no rack tile could simply be appended to an existing set. This forced real rearrangement.

**Why it was abandoned:**  
v1 could not reliably produce high-difficulty puzzles. The sacrifice approach is structurally random — there is no guarantee that the resulting puzzle is hard. The disruption score was the only difficulty lever, and it was blunt. The multi-metric evaluator was designed to fix this.

### v2: "Strategic Removal" Approach (phase4–phase7)

**Mechanism:**
1. `BoardBuilder` constructs a valid board with high tile overlap (tiles participate in many possible sets).
2. `TileRemover` removes tiles one at a time from the board, choosing tiles whose removal maximizes "cascade depth" (estimated rearrangement complexity).
3. Each removal is verified solvable with a short ILP timeout.
4. `DifficultyEvaluator` computes 8 metrics and a composite score (0–100).
5. Gates: reject if composite score is >1 tier off, if disruption is too low, if fragility is too low, or if a rack tile trivially extends a complete board set (`_any_trivial_extension_v2`).

**Parameters (phase7):**

```python
# Rack sizes per tier:
_RACK_SIZE_RANGES_V2 = {
    "easy": (2, 3), "medium": (3, 4), "hard": (4, 5),
    "expert": (5, 7), "nightmare": (6, 8),
}
# Board sizes per tier:
_BOARD_SIZE_RANGES_V2 = {
    "easy": (6, 9), "medium": (8, 11), "hard": (10, 13),
    "expert": (12, 15), "nightmare": (13, 16),
}
# Tier thresholds (composite score 0–100):
TIER_THRESHOLDS = {
    "easy": (0, 52), "medium": (38, 68), "hard": (55, 85),
    "expert": (68, 92), "nightmare": (75, 100),
}
```

## Root Cause Analysis

### Root Cause 1: The Metrics Measure Solver Complexity, Not Human Perception

The 8 difficulty metrics in `DifficultyEvaluator` were designed to approximate human difficulty but they actually measure properties of the ILP model:

| Metric | What it measures | Why it doesn't predict human difficulty |
|--------|-----------------|----------------------------------------|
| `branching_factor` | Avg valid placements per rack tile (ILP candidate sets) | Humans visually prune 95% of options instantly |
| `deductive_depth` | `chain_depth × log₂(branching_factor + 1)` | Artificially large when BF is high; BF is uncorrelated with human search effort |
| `red_herring_density` | Candidate sets that compete for solution-needed tiles | Humans don't enumerate candidate sets; they reason spatially |
| `working_memory_load` | Count of disrupted board sets in solution | Doesn't measure how hard it is to *find* the disruption |
| `tile_ambiguity` | Avg candidate sets per tile (board + rack) | Same problem as branching_factor |
| `solution_fragility` | Whether removing a rack tile changes the solution | Irrelevant to whether the human can find the solution at all |
| `disruption_score` | Board tiles moved to different sets | Measures solution complexity, not search complexity |
| `chain_depth` | Longest dependency chain in solution | Correct concept, but only 1 of 8 metrics; weighted 0.05 |

The composite formula is a weighted sum of normalised metrics. But none of these metrics actually predict how long a human will take to solve the puzzle, because human Rummikub difficulty is about *search* (finding the path), not *execution* (placing tiles once the path is known).

A puzzle with high branching_factor and high disruption_score can be solved instantly if there is one visually obvious move (place the rack tile that obviously extends an existing run, then the board rearranges itself). The ILP metrics will rate this as "expert" because the ILP had to explore many candidates. A human sees it in 5 seconds.

### Root Cause 2: TileRemover Removes Tiles From Known Solutions

The v2 approach is **epistemically backwards**:

1. Start with a valid board.
2. Remove tiles from it, tracking that the result is still solvable.
3. Present the result to the human.

The key problem: **the structural "fingerprint" of the original sets is still visible on the board**. When you remove tile T from set [A, B, T], the board retains [A, B] as a 2-tile stub. The player often sees [A, B] and immediately knows "T goes here." Even without the stub, the remaining board structure encodes the intended solution because it was a valid board before removal — the removal essentially shows the human the answer.

In v1, sacrificed sets are entirely absent. Their structure provides no visual hint. The player must reason from scratch.

In v2, removed tiles are "hidden" but their context survives. This is closer to a jigsaw with missing pieces than a Rummikub puzzle.

### Root Cause 3: `_any_trivial_extension_v2` Is Too Permissive

The original `_any_trivial_extension()` (v1) gates: *"reject if any rack tile can be appended to ANY board set."*

`_any_trivial_extension_v2()` gates: *"reject if any rack tile can be appended to any COMPLETE (≥3 tile) board set."*

The v2 version was introduced because sequential removal always creates 2-tile stubs, and the original gate rejected 100% of v2 puzzles. The fix was to ignore partial sets.

**But this means the filter now allows puzzles where rack tiles obviously complete 2-tile stubs** — which, in practice, is almost every v2 puzzle. The 2-tile stubs [A, B] are visually obvious slot-fillers. A human looks at the board, finds the stubs, and matches rack tiles to them in seconds. The filter was supposed to prevent this; the v2 variant defeats its own purpose.

Correct behaviour for v2 should be: reject if any rack tile has an obvious home *anywhere* — including completing partial sets, because a partial set is an even more obvious hint than a complete one (you can see it's missing something).

### Root Cause 4: Rack Sizes Are Too Small Relative to Board Complexity

Phase 7 expert: rack 5–7 tiles, board 12–15 sets (~36–50 tiles).

The human places 5–7 tiles on a board of 36–50 tiles. Even with high disruption score (many tiles moving), with such a small rack the player only needs to find the entry point. Once one tile is placed, the solver (the human) can often "pull the thread" and place the rest in a few seconds.

Phase 6 calibration batch (before phase 7, same results) showed expert puzzles taking 0.24–0.47 minutes. The rack sizes haven't changed between phases 6 and 7.

**v1 expert rack size: (6, 10).** Same range but more disruptive because the home sets were gone entirely. Disruption bands for v1 expert: min 32. v2 expert min disruption: 14. This is less than half.

### Root Cause 5: Normalization Ceilings Are Uncalibrated Guesses

The normalization ceilings in `difficulty_weights.json` determine at what raw value a metric reaches 1.0 (maximum contribution to the composite score):

```json
"normalization_ceilings": {
    "branching_factor": 40.0,   // a rack tile in 40+ candidate sets = maximum
    "deductive_depth":  16.0,
    "working_memory_load": 14.0, // 14 disrupted sets = maximum
    "tile_ambiguity":  32.0,
    "disruption_score": 50.0,   // 50 displaced tiles = maximum
    "chain_depth":  5.0
}
```

These numbers were set by hand and have never been validated against actual puzzle data. A nightmare puzzle scoring 80/100 composite reaches these ceilings — but reaching the ceiling doesn't mean the puzzle is hard for a human. It means the ILP model is large.

The phase7_batch_v1 nightmare puzzles have composite scores of 69–91. Yet they were solved in 17–49 seconds. The ceilings are simply wrong.

### Root Cause 6: The Calibration Signal Is Weak

**Phase 6 calibration:** 735 attempts for 5 puzzles — accumulated from months of development testing, not real human play. Data completely contaminated by developer sessions.

**Phase 7 calibration:** 1 run, 1 person (the developer who implemented the system). 25 puzzles. Average solve time 4–28 seconds across all tiers.

Thresholds in `_MIN_DISRUPTION_V2` and `_MIN_FRAGILITY_V2` were derived from the Phase 6 data:

```python
_MIN_DISRUPTION_V2 = {"easy": 3, "medium": 6, "hard": 10, "expert": 14, "nightmare": 18}
_MIN_FRAGILITY_V2  = {"easy": 0.0, "medium": 0.0, "hard": 0.1, "expert": 0.25, "nightmare": 0.4}
```

These thresholds do not represent human-perceived difficulty. They represent the developer's intuition about ILP metric values.

## What Was Attempted in Phase 7

Phase 7 was specifically designed to fix puzzle quality after Phase 6 showed all puzzles were trivially easy. The changes made:

1. **`_any_trivial_extension_v2()`** — added trivial-extension gate to v2 path (only checked complete sets).
2. **`_solve_timed()` thread wrapper** — fixed HiGHS hanging in AnyIO worker threads on Windows.
3. **Revised `red_herring_density`** — changed from "any non-solution set" to "sets competing for solution-needed tiles." This was a correct methodological fix but the metric still doesn't predict human difficulty.
4. **Revised tier thresholds** — from `(0,20)/(15,35)/...` to `(0,52)/(38,68)/...`. Wider bands allow more puzzles through; didn't fix the underlying problem.
5. **Raised normalization ceilings** — `branching_factor` from 8→40, `deductive_depth` from 10→16, etc. This lowered composite scores for the same raw data, making fewer puzzles reach the top tiers artificially. Again a calibration fix for the scorer, not the generator.
6. **`batch_run_id` isolation** — fixed telemetry contamination between sessions. Correct fix.
7. **`--fit-weights` in calibrate.py** — linear regression of solve_time on metrics. Not yet used; requires 20+ solved sessions per tier.

**None of these changes addressed the two structural problems:** (a) metrics are solver-centric not human-centric, and (b) v2 generation leaves the solution's structure visible on the board.

Phase 7 calibration result: identical to Phase 6 in terms of human experience. 0 undos, ~30s solve times across all tiers.

| `backend/solver/generator/calibrate.py` | Analysis tool: `--batch`, `--stats`, `--fit-weights` modes |
| `backend/solver/generator/gen_calibration_batch.py` | Generates a validated batch of puzzles for calibration |

### v1 vs v2 selection

`generate_puzzle()` in `puzzle_generator.py` always calls `_attempt_generate_v2()`. v1 is still present in the file but is not called for live or pool generation. The pool pregeneration script (`pregenerate.py`) also uses v2.

### The ILP solver

The solver (`solver/engine/solver.py`) uses HiGHS via `highspy`. It formulates tile placement as an integer linear program: decision variables are which candidate sets to activate; objective is to maximize rack tiles placed. Solve time: <100ms typical, up to 30s for complex configurations. On Windows in AnyIO worker threads, HiGHS sometimes ignores its time_limit — mitigated by `_solve_timed()` daemon-thread wrapper in `tile_remover.py`.

## Hypotheses for a Real Fix

The following are hypotheses only — they should be validated empirically:

### Hypothesis A: Return to v1 Sacrifice + Better Difficulty Filtering

The v1 "sacrifice" approach conceptually prevents obvious solutions: the sacrificed sets' homes are gone. The problem was that v1 couldn't reliably produce high-difficulty puzzles within reasonable attempt counts. However, v1 was never combined with a properly calibrated multi-metric evaluator. A possible direction:

- Keep v1's sacrifice mechanism (entire sets removed, not individual tiles).
- Apply the multi-metric evaluator to select which sacrifice results are genuinely hard.
- Use empirical calibration (real human data, >50 people, >100 puzzles per tier) to set thresholds.
- The `_any_trivial_extension` gate is correct as-is for v1 (rejects any rack tile that can trivially extend any set).

### Hypothesis B: Force Non-Obvious Placements at Generation Time

Instead of measuring difficulty after the fact, build difficulty constraints into the generation process:

- For each rack tile, require that there is NO visually obvious placement (no set on the board that the tile extends without displacing other tiles).
- Use constraint propagation: a placement is "obvious" if it can be decided in isolation; it's "non-obvious" if it requires reasoning about at least K other tiles.
- This is harder to compute but ensures difficulty by construction.

### Hypothesis C: Search-Space-Based Metrics

Replace the current 8 metrics with metrics that directly model the human search space:

- **Dead-end density:** What fraction of first-move attempts are dead ends (cannot lead to full placement)? Requires running the solver N times with forced first moves.
- **Minimum backtrack depth:** How many wrong placements must the human undo before finding the solution? Computable via bounded DFS from the human's perspective.
- **Forced-move ratio:** What fraction of moves in the solution are forced (only one valid option)? Low = easier (fewer decisions). High = harder (many forced steps, easy to follow once you find the first move).

These require more computation but directly predict human solve time.

### Hypothesis D: Use Real Human Data to Train the Scoring Model

The `--fit-weights` mode in `calibrate.py` implements linear regression of solve_time on the 8 metrics. This has never been used on real data because the calibration pool only has developer sessions. With:

- 30+ real human players
- 50+ puzzles per tier
- Multiple sessions per player

…the regression might find that only 1–2 of the 8 metrics actually predict solve time, and identify the correct normalization ceilings. The rest of the metrics may be pruned. This is the most empirically grounded approach but requires actual user research.

### Hypothesis E: Puzzle Types With Known Difficulty Properties

Some Rummikub puzzle configurations are known to be hard by structure, independent of metrics:

- **Joker replacement:** A joker on the board must be displaced and used elsewhere. High search complexity because the joker's position affects every set it touches.
- **Multi-set chains:** A tile can only be placed if another tile is first moved to a set that requires a third tile to be moved out — the chain has no shortcut.
- **False runs:** The board contains runs/groups that look like they should be broken but breaking them leads to dead ends.

Generating puzzles specifically with these structural properties (rather than scoring random puzzles after the fact) may be more reliable.
